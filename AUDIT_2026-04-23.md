# koivulabs-hub Audit — 2026-04-23

## Yhteenveto
Auditoitu Next.js 15 + React 19 + Firebase -stack (luettu ~20 avaintiedostoa). `tsc --noEmit` = 0 virhettä, `npm run lint` = 0 virhettä / 28 varoitusta, `npm run build` menee läpi (epätodennäköinen `.nft.json`-viimeistelyvirhe ei estä buildia). Löydökset: **4 kriittistä, 6 keskitasoa, 5 parannusta** — painopiste Firestore-sääntöjen puuttumisessa, binääritiedoston commitissa, webhook-secretin kaksoiskäytössä admin-reittien portinvartijana ja middlewaren puuttuvassa kryptografisessa tokenin varmistuksessa.

## 🔴 Kriittiset (korjaa heti)

### K-1: `firestore.rules` puuttuu kokonaan repositoriosta — Turso
- **Tiedosto:** `firestore.rules` (ei olemassa repossa; hakuosuma vain `node_modules`:ssa)
- **Ongelma:** Repossa ei ole `firestore.rules`-tiedostoa eikä `firebase.json`:ia. Säännöt elävät vain Firebase-konsolissa → ei versiohallintaa, ei review-pipelineä, ei rollbackia. CLAUDE.md vaatii eksplisiittisesti että "Firestore security rules tarkistetaan ennen jokaista deployta". Tämä on mahdotonta ilman että säännöt ovat koodissa.
- **Riski:** Client-SDK luetaan `projects`- ja `logs`-kokoelmia suoraan selaimesta (`src/lib/projectService.ts:17`, `src/lib/logService.ts:53`). Jos säännöt ovat `allow read, write: if true` (yleinen "toimii ensin" -oletus), kuka tahansa pystyy kirjoittamaan/poistamaan Firestoreen. Admin-panelin `deleteProject`/`deleteLog` ajetaan client-SDK:lla, joten kirjoitusoikeudet **joudutaan** avaamaan jollekin tasolle. Ainoa tieto sääntöjen tilasta on `src/app/now/page.tsx:47` -kommentti "Learning: Firebase security rules. In production. The hard way." — ei todistetta laadusta. Lisäksi `pendingPosts`, `userState`, `invoices`, `timeEntries`, `clients`, `siteContent` -kokoelmat sisältävät PII:tä (Y-tunnus, IBAN, sähköpostit, puhelin, hintatiedot) ja ovat yhdessä ja samassa projektissa.
- **Korjaus:** (1) Vedä nykyiset säännöt alas: `firebase init firestore` + `firebase deploy --only firestore:rules` jotta ne committautuvat. (2) Deny-by-default: `projects` ja `logs` → `allow read: if resource.data.status == "Published"` + `allow write: if false` (kaikki kirjoitukset Admin SDK:n kautta). (3) `pendingPosts`, `userState`, `invoices`, `timeEntries`, `clients`, `siteContent` → `allow read, write: if false` (kosketetaan vain Admin SDK:sta). (4) Lisää `firestore.rules` + `firebase.json` gittiin ja pakollinen review deploy-checklistiin.
- **Työmäärä:** M

### K-2: `koivulabs.rar` committattu repositorioon — Turso
- **Tiedosto:** `koivulabs.rar` (repo-juuri, 122 KB, `git log` näyttää sen trackattuna)
- **Ongelma:** Projektin juuressa on 122 KB rar-arkisto. `.gitignore`-rivi `*.txt` kattaa vain tekstit — `.rar`/`.zip`/`.7z` eivät ole ignoressa. Tiedostoa ei voi tarkastaa paikan päällä sisällöstä ilman purkamista, mutta nimi ja sijainti viittaa varmuuskopioon tai "dumppiin". CLAUDE.md:n pääsääntö: "Service account JSON-tiedostot eivät koskaan repositorioon" + strict_donts.md #6 — arkiston sisältöön kohdistuu sama riski.
- **Riski:** Mahdollinen secrets-vuoto (service account JSON, `.env.local`, credentials backup). Jo push:attu GitHubiin — jos sisältö on sensitiivistä, git history on jo saastunut ja pelkkä `git rm` ei riitä. Lisäksi 122 KB binaari turhaan bundlessa.
- **Korjaus:** (1) Pura `koivulabs.rar` paikallisesti ja tarkasta sisältö ennen jatkoa. (2) Jos sisältää avaimia → kierrätä KAIKKI avaimet välittömästi (Firebase service account, OpenAI, Telegram bot, GitHub PAT) ja puhdista git-historia `git filter-repo`:lla. (3) Lisää `.gitignore`:en `*.rar`, `*.zip`, `*.7z`, `*.tar.gz`, `*.bak`. (4) `git rm koivulabs.rar && git commit`.
- **Työmäärä:** S (jos sisältö OK) / L (jos vuotaa avaimia)

### K-3: Middleware hyväksyy MINKÄ TAHANSA JWT-muotoisen merkkijonon admin-evästeeksi — Turso
- **Tiedosto:** `src/middleware.ts:7`
- **Ongelma:** Admin-reitin ainoa gate:
  ```ts
  if (!session || !session.startsWith('eyJ')) {
      return NextResponse.redirect(new URL('/admin/login', request.url));
  }
  ```
  `eyJ` on pelkkä base64-prefix `{"` — mikä tahansa string joka alkaa `eyJ` läpäisee tarkistuksen. Firebase ID-tokenia ei verifioida `admin.auth().verifyIdToken()`:llä, allekirjoitusta ei tarkisteta, voimassaoloaikaa ei tarkisteta, uid/email-whitelistiä ei ole.
- **Riski:** **Kenen tahansa** `document.cookie = "__koivu_session=eyJfoobar; path=/"` selaimessa antaa pääsyn `/admin`-reittiin. Admin-sivu ohjaa clienttisivun Firebase-tilan kautta uudelleen — mutta sivu RENDERÖITYY silti hetkeksi, API-routet jäävät täysin suojaamatta (ei middleware-matchia `/api`:lle), ja Firestore-writet kulkevat client-SDK:sta ilman auth-tarkistusta (ks. K-1).
- **Korjaus:** (1) Vaihda middleware käyttämään `firebase-admin.auth().verifySessionCookie(session, true)` — vaatii että `/api/auth/session` vaihtaa ID-tokenin session cookieen (`auth.createSessionCookie()`). (2) Tarkista lisäksi uid/email on whitelistattu (admin-käyttäjä). (3) Middleware ei toimi Edge runtimessa `firebase-admin`:n kanssa — joko siirrä admin-tarkistus `layout.tsx`-server componentiksi tai käytä lightweight JWT-verifyä (jose) Google:n JWKS:lla.
- **Työmäärä:** M

### K-4: `/api/admin/*` käyttää samaa secrettiä kuin Telegram-webhook, välitetään query-parametrina — Turso
- **Tiedosto:** `src/app/api/admin/fix-slugs/route.ts:19`, `src/app/api/admin/sync-projects/route.ts:9`
- **Ongelma:**
  ```ts
  const secret = req.nextUrl.searchParams.get('secret');
  if (secret !== process.env.TELEGRAM_WEBHOOK_SECRET) { ... }
  ```
  (1) Kaksi täysin erillistä trust-domainia (Telegram IPC + admin-migraatiot) jakavat saman shared secretin — jos joku vuotaa, molemmat kaatuvat. (2) Secret kulkee **query-stringissä** → päätyy Vercel accesslogeihin, selaimen historyyn, browser referrer -headeriin, CDN-lokeihin. (3) Endpoint on GET — voidaan laukaista pelkällä linkin klikkauksella ilman CSRF-suojaa. (4) Molemmat reitit käyttävät Admin SDK:ta ja yliajavat Firestoren — mutaatio-endpoint GET:ssä rikkoo REST/HTTP-semantiikan ja tekee idempotenssista kevyen.
- **Riski:** Secret vuotaa helposti (Vercel-paneelin logit, jaetut linkit, selain-historia). Vuodon jälkeen hyökkääjä voi (a) pilkata Telegram-webhookiin mitä tahansa sisältöä — botti auto-julkaisee logbook-postauksia ja commitoi GitHub:iin, (b) kutsua `fix-slugs`-reittiä korruptoiden Firestore `logs`-dokumentit, (c) ylikirjoittaa `projects/*` syncillä.
- **Korjaus:** (1) Erota secrettit: `ADMIN_API_SECRET` ≠ `TELEGRAM_WEBHOOK_SECRET`. (2) Vaihda `GET` → `POST` ja lue secret `Authorization: Bearer`-headerista, ei URL:stä. (3) Lisää IP-allowlist tai sido reitit middleware-session-evästeen taakse. (4) Kun admin-auth on korjattu (K-3), käytä sitä — erillinen secret-avain on band-aid.
- **Työmäärä:** S

## 🟡 Keskitaso

### M-1: `/api/contact` vastaanottaa liidit mutta ei lähetä mitään eteenpäin — Louhi
- **Tiedosto:** `src/app/api/contact/route.ts:13`
- **Ongelma:** Endpoint logittaa vain `console.log` ja palauttaa `{ ok: true }`. Ei email-notifikaatiota, ei Telegram-ping:iä, ei DB-tallennusta. TODO-kommentti rivillä 12.
- **Riski:** Kaikki koivulabs.com/contact-liidit katoavat Vercel-lokeihin (säilyvät vain ~1 viikko). Rahaa jää pöydälle. Sukulaisuus mistakes_log SÄÄNNÖLLE 22: "endpoint ilman notifikaatiota = hiljainen fail".
- **Korjaus:** Tallenna Firestoreen `contactMessages`-kokoelmaan ja ping Telegramiin (sama BOT_TOKEN + MY_TELEGRAM_USER_ID kuin webhookissa). Lisää email-fallback Resendillä.
- **Työmäärä:** S

### M-2: In-memory rate limit ei toimi Vercel serverlessissä — Turso/Tapio
- **Tiedosto:** `src/app/api/metadata/route.ts:3`, `src/app/api/refine/route.ts:3`
- **Ongelma:** `const rateLimitMap = new Map()` on per-lambda-instance. Vercel Functions skaalaa horisontaalisesti → 10 samanaikaista kylmäkäynnistystä = 10× 10 pyyntöä/min = 100 todellisessa rajassa. Lisäksi `x-forwarded-for` voi olla lista, IP-spoofaus on triviaali.
- **Riski:** OpenAI API -kustannusten pakottaminen toisen maksettavaksi: endpointit kutsuvat `gpt-4o`:ta ilman authia eikä todellista rajoitusta. Perus DoS + tilikustannus.
- **Korjaus:** (1) Upstash Rate Limit tai Vercel KV (persistoitu). (2) Aseta kova OpenAI-budjettiraja dashboardissa. (3) Lisää auth (sama Firebase session cookie) tai CAPTCHA jos julkiseen käyttöön.
- **Työmäärä:** M

### M-3: `listTimeEntriesByInvoice` hakee aina 100 riviä ja filtteröi muistissa — Louhi/Ilmarinen
- **Tiedosto:** `src/lib/firestoreRest.ts:361`
- **Ongelma:** `db().collection('timeEntries').limit(100).get()` ilman `where('invoiceId', '==', invoiceId)`. Kun kirjauksia tulee >100, vanhimmat eivät koskaan päädy laskun PDF:ään — hiljainen data-vuoto laskutuksessa. Sama patologia `getOrCreateOpenInvoice`:ssa (`listInvoices('open')` palauttaa max 50 ja suodattaa muistissa → jo 50 laskun jälkeen uusi open-invoice luodaan vaikka yksi on olemassa).
- **Riski:** Laskutus jakautuu virheellisiin invoice-dokumentteihin. Asiakaskokemus rikki ja Y-tunnus-kirjanpito sekaisin.
- **Korjaus:** Käytä Firestore-queryjä: `collection('timeEntries').where('invoiceId', '==', invoiceId).get()` + vastaava `listInvoices`. Poista `limit(100)` tai aseta se selvästi suurempi.
- **Työmäärä:** S

### M-4: CSP sallii `unsafe-eval` ja `unsafe-inline` skripteille — Turso
- **Tiedosto:** `vercel.json:26`
- **Ongelma:** `script-src 'self' 'unsafe-eval' 'unsafe-inline'`. Tämä käytännössä kumoaa CSP:n XSS-suojan. `unsafe-eval` on Next.js dev-tilalle, mutta tuotannossa tarpeeton React 19 + ei Reflect-polyfillejä.
- **Riski:** XSS payload (esim. logbook-postauksen markdown-sisällöstä jos joku saa publishing-oikeuksia) voi suorittaa `eval`/inline-skriptiä suoraan. Yhdistettynä K-1:n avoimiin Firestore-sääntöihin tämä on suora attack chain.
- **Korjaus:** Vaihda nonce-pohjainen CSP: Next.js middleware injektoi `nonce` jokaiseen scriptiin, CSP = `script-src 'self' 'nonce-xyz' 'strict-dynamic'`. Vähintään poista `unsafe-eval`.
- **Työmäärä:** M

### M-5: Telegram-webhookin "fire-and-forget" `console.error`:eissa katkaisee root cause -jäljityksen — Louhi
- **Tiedosto:** `src/app/api/webhooks/telegram/route.ts:1598`
- **Ongelma:** Webhook palauttaa AINA `200 OK` riippumatta siitä kaatuuko `handleMessage`/`handleCallbackQuery`. Virheet pelkästään logitetaan. Tämä on *hyvä* Telegram-retry:n estämiseksi, mutta yhdistettynä siihen että `handleMessage` itse ei throwaa (sisäinen try-catch), ulkoiselle seurannalle ei jää signaalia.
- **Riski:** Hiljaiset tallennusvirheet julkaisuputkessa — pendingPost savePendingPost kaatuu → käyttäjä ei tiedä.
- **Korjaus:** Lisää Sentry/Axiom-integraatio tai vähintään Telegram-viesti käyttäjälle kun kriittinen polku kaatuu (saveLogToFirestore, commitToGitHub). Log-tägi (`[koivu-voice]`) on hyvä alku mutta ei riitä havainnointiin.
- **Työmäärä:** S

### M-6: `useEffect` puuttuva dependency aiheuttaa stale data -riskin admin-panelissa — Louhi
- **Tiedosto:** `src/app/admin/page.tsx:37-39`
- **Ongelma:** `useEffect(() => { loadData(); }, [activeTab]);` — `loadData` suljetaan closureen eikä ole deps-listassa. Toimii tässä koska `loadData` on stabiili nimi, mutta ESLint varoittaa ja React 19 strict-modessa tämä voi triggeröidä tuplarender-latausta.
- **Riski:** Admin-paneli voi näyttää vanhentunutta dataa editoinnin jälkeen jos `loadData`:n riippuvuudet muuttuvat.
- **Korjaus:** `useCallback(loadData, [activeTab])` ja lisää deps-listaan. Pidemmällä aikavälillä: React Query tai SWR.
- **Työmäärä:** S

## Rajoitteet
- `npm audit` ei ajettu: oletusarvo "ei ajeta ilman eksplisiittistä pyyntöä" tässä read-only-skannauksessa. Suositus: aja `npm audit --production` ennen deployta.
- `gh api repos/koivulabs/koivulabs-hub/dependabot/alerts` ei testattu (ei gh-autentikointia käytettävissä tässä istunnossa).
- `ctx-scan-repo`/`ctx-toolbox` ei ajettu (rajattu aika, painopiste koodi-auditointiin).
- `firestore.rules`:n todellista tilaa Firebase-konsolissa ei voida tarkistaa repositoriosta — K-1 perustuu siihen että tiedostoa ei ole repossa. Voi olla että säännöt ovat konsolissa OK, mutta se ei ole versioitua eikä reviewattavaa.
- `koivulabs.rar`-binäärin sisältöä ei purettu (vaatii user-action — ks. K-2).
- `npm run build` tuotti yhden `ENOENT ... _not-found/page.js.nft.json` -virheen `Collecting build traces` -vaiheessa. Ei estä staattista build-tulostetta (39 sivua generoitu), mutta voi vaikuttaa Vercel-deploy-traceihin. Ei tutkittu juurisyytä tässä auditissa.
- Komponenttikerros (`src/components/**`) käyty läpi vain grep-tasolla (`dangerouslySetInnerHTML`, `catch {}`) — syvempi UI-tarkastus vaatisi Ainon näkökulman erillisen ajon.
